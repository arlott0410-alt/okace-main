# ย้ายกะอัตโนมัติ + ผลกระทบต่อการจองวันหยุด

## 1) กะเปลี่ยนทันทีเมื่อถึงวันที่ตั้งไว้

- **ตารางย้ายกะ/สลับกะ** (shift_swaps, cross_branch_transfers) กำหนด `start_date` และ `end_date` — ภายในช่วงนี้พนักงานถือว่าอยู่กะ/แผนกปลายทาง
- **ในแอป (ตารางวันหยุด, โควต้า, จองวันหยุด):** ระบบใช้ **กะ/แผนกที่มีผลในวันนั้น** (effective) ไม่ยึดแค่ค่าในโปรไฟล์  
  - ตัวอย่าง: ตั้งย้ายกะวันที่ 15 มี.ค. → ตั้งแต่ 15 มี.ค. เป็นต้นไป ระบบนับโควต้าและจองวันหยุดตามกะปลายทาง
- **ให้โปรไฟล์ (default_shift_id / default_branch_id) อัปเดตเมื่อถึงวันนั้น:**
  - ฟังก์ชัน `apply_scheduled_shift_changes_for_date(วันนั้น)` จะอัปเดต `profiles.default_shift_id` และ `default_branch_id` ตามรายการย้ายกะที่ approved และวันที่อยู่ในช่วง [start_date, end_date] (end_date NULL = ถาวร)
  - **ถ้าวันเปลี่ยนกะแล้วแต่กะบนแดชบอร์ดยังไม่เปลี่ยน** แปลว่า **ยังไม่ได้รันฟังก์ชันนี้ในวันนั้น** — ต้องทำอย่างใดอย่างหนึ่ง:
    1. **ตั้ง Supabase Cron** ให้รันทุกวัน (แนะนำ): ดูไฟล์ `supabase/sql/cron_apply_shift_changes_setup.sql` — เปิด pg_cron แล้ว schedule job เรียก `run_apply_shift_changes_and_log()` ทุกวัน 00:01 น. (Asia/Bangkok)
    2. **รันมือใน SQL Editor:** `SELECT apply_scheduled_shift_changes_for_date(current_date);` หรือ `SELECT run_apply_shift_changes_and_log();`
    3. **กดปุ่มในแอป:** ตั้งค่า → ส่วน "ย้ายกะอัตโนมัติ" → กด **อัปเดตกะตามกำหนดวันนี้** (เฉพาะ Admin / ผู้จัดการ / หัวหน้าผู้สอน) แล้วให้พนักงานรีเฟรชหรือออก/เข้าสู่ระบบ

## 2) ถ้าลงวันหยุดแล้วพนักงานโดนย้ายกะ — กระทบ flow / กติกาจองวันหยุดอย่างไร

- **รายการวันหยุด (holidays) ไม่ถูกลบหรือเปลี่ยนอัตโนมัติ** เมื่อมีการย้ายกะ — วันที่หยุดยังคงอยู่
- **กติกาโควต้า:** ระบบนับโควต้าตาม **กะ/แผนกที่มีผลในวันนั้น**:
  - **วันที่มีการจองวันหยุด:** ใช้กะ/แผนก **effective ในวันนั้น** ว่าอยู่กลุ่ม (ขอบเขต) ไหน → นับโควต้าในกลุ่มนั้น
  - ตัวอย่าง: จองวันหยุดวันที่ 20 มี.ค. ตอนนั้นอยู่กะกลาง → หลังย้ายกะมีผล 15 มี.ค. ไปกะเช้า แล้ววันที่ 20 มี.ค. ระบบถือว่าอยู่กะเช้า → โควต้าที่ใช้ไปในวันนั้นจะนับใน **กลุ่มกะเช้า** ไม่ใช่กะกลาง
- **การจองวันใหม่:** ตอนกดจอง ระบบจะใช้กะ/แผนก effective ใน**วันที่จอง**เป็นขอบเขตตรวจโควต้า และบันทึก `branch_id` / `shift_id` ตาม effective นั้น
- **สรุป:** ลงวันหยุดแล้วโดนย้ายกะ = วันหยุดยังอยู่ แต่โควต้าที่นับจะสอดคล้องกับกะ/แผนกที่มีผลในวันนั้น (ไม่พังกติกา จองและนับถูกกลุ่ม)

## 3) กะดึกย้ายไปกะเช้า/กลาง — พิเศษกว่ากะอื่น

- **กะดึกเลิกงานเป็นวันถัดไป** ดังนั้นถ้าวันที่ตั้งเปลี่ยนกะ (start_date) เป็นวันเปลี่ยนกะดึก→เช้า/กลาง ระบบถือว่า **วันนั้นเป็นวันหยุดในตัว** — พนักงานไม่มาทำงานในวันนั้น (จะเข้างานกะปลายทางในวันถัดไป)
- **ในตารางวันหยุด / โควต้า:**
  - **วันเปลี่ยนกะ (start_date) สำหรับดึก→เช้า/กลาง:** ระบบยังแสดง effective = **กะดึก** ในวันนั้น แต่ **ไม่นับคนนี้ในโควต้า** (ไม่นับเป็น “มาทำงาน”) — เหมือนวันหยุดในตัว
  - **ตั้งแต่ start_date + 1 เป็นต้นไป:** effective = กะปลายทาง (เช้า/กลาง) ตามปกติ
- **กะอื่น (เช้า↔กลาง ฯลฯ):** วันเปลี่ยนกะสามารถสลับ/ย้ายได้ในวันนั้นเลย และ effective = กะปลายทางในวันนั้น

---

## 4) วันหยุด/วันลากับโควต้า และก่อน/หลังย้ายกะ

- **ทุกการลงวันหยุด/วันลา** มีผลต่อโควต้าในกติกาที่นับ “จำนวนคน” — เพราะวันลาถือว่าพนักงานไม่มาทำงานในวันนั้น
- **ถ้าวันลาเกิดก่อนที่หัวหน้าจะลงวันหยุด/วันลา:** ไม่มีปัญหา ระบบนับตามปกติ
- **ถ้ายังไม่เกิด (ยังไม่ถึงวันนั้น) แต่หัวหน้าลงวันหยุด/วันลาก่อน:** ระบบจะ **คำนวณว่าวันนั้นมีคนนี้หยุด** อยู่แล้ว — ตอนนับโควต้าวันนั้นจะนับคนนี้เป็น “ใช้โควตา” ตามกติกา (กะ/แผนกที่มีผลในวันนั้น)
- **วันเปลี่ยนกะ:** แสดงบนตารางวันหยุดเหมือนการเปลี่ยนกะในจัดการสมาชิก แต่เป็นการตั้งเวลาไว้ และกะดึกจะพิเศษดังข้อ 3
